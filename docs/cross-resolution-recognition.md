# 跨分辨率识别与坐标映射说明

本文说明可可萝自动会战助手当前如何在不同 Android 分辨率、宽高比和 DPI 环境中定位游戏 HUD、识别时钟/TP/控制状态，并让自动点击与识别使用同一套坐标。

当前实现以 `1920×1080` 为唯一参考坐标，不为每一种设备分辨率维护独立坐标表。

## 总体流程

每个截图帧依次经过：

1. 从 `MediaProjection` 取得实际像素尺寸的横屏图像。
2. 根据实际宽高计算一个等比游戏视口。
3. 按中央或右侧锚点映射参考 ROI。
4. 使用战斗开始按钮和菜单模板校准水平偏移。
5. 裁剪时钟、TP、AUTO、SET、菜单等区域。
6. 对裁剪图进行模板、数字结构或颜色长度识别。
7. 使用战斗会话门控、时序过滤和控制状态过滤决定结果是否可用于自动化。
8. 自动点击通过同一个坐标映射器计算实际位置。

## 1920×1080 参考坐标

识别区域定义在 `BattleReferenceRegions` 中：

| 区域 | 参考矩形 `(x, y, width, height)` | 锚点 |
|---|---|---|
| 时钟 | `(1619, 38, 64, 27)` | 右侧 |
| 战斗开始按钮 | `(1565, 850, 275, 115)` | 中央 |
| 加载区域 | `(1545, 955, 190, 60)` | 右侧 |
| 菜单 | `(1761, 33, 87, 37)` | 右侧 |
| 全局 SET | `(1788, 644, 87, 86)` | 右侧 |
| AUTO | `(1783, 795, 95, 78)` | 右侧 |
| 角色 SET | `x = 540 + 240 × 角色序号`，`y = 761`，`74×73` | 中央 |
| TP HUD | `(384, 1034, 1160, 25)` | 中央 |

点击参考点同样基于 `1920×1080`：

- 菜单、AUTO、全局 SET 使用右侧锚点。
- 五名角色使用中央锚点。

## 等比视口计算

实际截图宽高记为 `W`、`H`，参考宽高为 `1920×1080`。

```text
scale = min(W / 1920, H / 1080)
spareX = max(0, W - 1920 × scale)
offsetY = max(0, (H - 1080 × scale) / 2)
```

使用 `min()` 的目的是保持参考画面等比缩放，不根据整个屏幕宽度强制拉伸参考坐标。

常见示例：

| 实际分辨率 | scale | spareX | 含义 |
|---|---:|---:|---|
| `1920×1080` | `1.000` | `0` | 与参考完全一致 |
| `2800×1264` | 约 `1.170` | 约 `553px` | 按高度缩放，左右存在额外宽度 |
| `2880×900` | 约 `0.833` | `1280px` | 极端超宽，锚点差异被显著放大 |

## 中央和右侧双锚点

额外横向空间不能统一分配，因为游戏不同 HUD 的布局规则不同。

### 中央锚点

```text
x = spareX / 2 + calibrationCenterX + referenceX × scale
```

用于：

- 战斗开始按钮；
- 五名角色和角色 SET；
- TP 能量条；
- 角色点击。

### 右侧锚点

```text
x = spareX + calibrationRightX + referenceX × scale
```

用于：

- 时钟；
- 菜单；
- 加载区域；
- AUTO；
- 全局 SET；
- 对应点击。

纵向统一使用：

```text
y = offsetY + referenceY × scale
```

## 水平模板校准

公式映射只能处理标准等比布局，无法完全覆盖黑边、安全区和游戏自身的细微偏移。因此每次新战斗会重置校准状态，并进行两次模板搜索。

### 中央校准

- 使用战斗开始按钮模板。
- 在公式位置左右 `±180px` 范围搜索。
- 先按 `12px` 步长粗搜索，再在最佳位置附近逐像素搜索。
- 模板得分达到 `0.72` 后保存中央偏移。

### 右侧校准

- 使用菜单按钮模板。
- 在公式位置左右 `±120px` 范围搜索。
- 同样使用粗搜索加逐像素精搜索。
- 菜单得分达到 `0.70` 后保存右侧偏移。

同一场战斗中，中央区域共享中央偏移，右侧区域共享右侧偏移。识别与点击都会使用保存后的偏移。

当前只校准水平位置，不校准垂直位置、ROI 宽高或每个控件的独立偏移。

## ROI 裁剪与模板归一化

`ImageRoiExtractor` 将参考矩形的左上角和右下角分别映射到实际截图，再直接从 `Image` 像素缓冲中提取 `PixelImage`。

不同分辨率下，实际裁剪尺寸可能不同。固定模板匹配时会把裁剪图按模板尺寸重新采样，再计算亮度相关度。因此正常的等比放大或缩小不需要为每个分辨率维护独立模板。

## 时钟识别

时钟使用右侧锚点，参考 ROI 为 `(1619, 38, 64, 27)`。

识别器会：

1. 从 ROI 中分割分钟、秒十位和秒个位。
2. 分别计算数字候选分数。
3. 组合为 `M:SS`。
4. 生成原始置信度和替代候选。
5. 交给 `RecognitionFilter` 做时序验证。

生产最低置信度为 `0.75`。过滤器还会检查：

- 时间范围必须为 `0:01～1:30`；
- 时间不能无故回升；
- 跳秒幅度必须符合经过时间；
- 大幅下降需要确认；
- 上一秒的替代候选可用于补救单帧主候选错误。

悬浮窗有时会显示正在变化的原始 OCR 文本，但只有通过过滤器的时间才会用于轴调度。

## TP 能量识别

TP 使用中央锚点裁剪整个能量 HUD，再按相对位置划分五名角色的子区域。

识别器不读取文字，而是逐列判断蓝色像素占比，通过最后一个有效蓝色列估算填充比例：

```text
TP 比例 = 已填充到的最后一列 / 能量区域总宽度
```

当前阈值：

- TP 比例达到 `0.97` 视为满能量。
- 上一帧满能量、当前帧低于 `0.30` 时视为角色释放一次 UB。

由于 TP 依赖颜色和长度，通常比数字模板更能容忍轻微尺寸变化，但仍依赖能量 HUD 的裁剪位置正确。

## AUTO、全局 SET 和角色 SET

AUTO 与全局 SET 使用开启/关闭模板成对比较：

- 最高模板分数必须达到最低值；
- 开启与关闭分数差必须达到最低 margin；
- 否则状态为 `UNKNOWN`。

角色 SET 图标存在呼吸动画，匹配器会搜索多种模板尺度和少量局部位移。青蓝颜色只能作为辅助证据，不能仅因为背景青蓝覆盖率较高就直接判断开启；还必须具备 SET 图标结构相关度。

## 控制状态的跨帧过滤

AUTO 和 SET 不会根据单帧结果立即改变状态。

`BattleControlObservationFilter` 区分：

- `TRUSTWORTHY`：可信稳定状态；
- `PENDING_CONFIRMATION`：新状态等待连续帧确认；
- `RAW_UNTRUSTWORTHY`：原始观察存在未知状态或不一致；
- `IMPLAUSIBLE_TRANSITION`：不合理的批量跳变；
- `MISSING`：裁剪或识别缺失。

短暂不可信时自动化会冻结新的控制点击，并保留上一个稳定状态。连续 8 帧仍不可信时才进入菜单安全暂停。可信帧恢复后计数立即清零。

## 战斗会话门控

自动化不会因为画面中出现一个时间文本就立即运行。

```text
WAIT_START → WAIT_LOADING → WAIT_CLOCK → RUNNING
```

- `WAIT_START`：等待战斗开始按钮模板。
- `WAIT_LOADING`：等待加载模板，或使用可信菜单/战斗 HUD 兜底。
- `WAIT_CLOCK`：等待有效且可信的开场 `1:30`。
- `RUNNING`：允许时钟调度、TP 触发、控制确认和自动点击。

等待有效 `1:30` 时，当前界面会显示原始时钟文本，但不会把原始时间用于调度。TP 检测仍会执行并写入诊断文件，但早退状态文本目前没有显示 TP。

## 点击与识别共享坐标

`ActionExecutor` 不维护另一套屏幕坐标。所有点击点通过 `GameCoordinateMapper` 映射，并使用与识别相同的中央/右侧锚点及校准偏移。

这样可以避免：

- ROI 已适配新分辨率，但点击仍落在旧坐标；
- 菜单识别发生偏移后，菜单点击没有同步偏移；
- 超宽屏角色区域和右侧控制区域使用同一种错误偏移。

## DPI 的作用

坐标映射器本身不直接读取 DPI，只使用截图的实际像素宽高。

DPI 会间接影响：

- Android `dp` 控件和助手悬浮窗的实际像素尺寸；
- 系统栏、安全区和资源密度选择；
- 部分游戏或虚拟显示的布局尺度。

因此相同分辨率、不同 DPI 仍可能产生不同的最终截图布局，正式回归应同时覆盖目标 DPI 范围。

## 当前验证结果

已经确认正常的分辨率包括：

- `1920×1080 / DPI 326`；
- `2400×1080 / DPI 326` 的完整成功会话；
- `2600×1200 / DPI 326`；
- `2800×1264 / DPI 326`。

`2800×1264` 后续已切换到 DPI `450`，用于覆盖目标 DPI 范围。

极端压力测试 `2880×900 / DPI 450` 暴露了当前模型的边界：游戏内时钟肉眼观察仍保持正常比例和合理位置，但秒十位数字的诊断分数明显下降，出现大量 `low-confidence` 和 `clock-split-failed`。

正常与极宽会话中相同 `1:04` 的数字分数大致为：

| 分位 | 正常 `2800×1264` | 极宽 `2880×900` |
|---|---:|---:|
| 分钟 | `0.786` | `0.750` |
| 秒十位 | `0.669～0.724` | `0.537～0.565` |
| 秒个位 | `0.779` | `0.709～0.732` |

主要退化集中在秒十位，目前更可能是时钟 ROI、数字分位裁剪或菜单与时钟共享右侧校准产生了细微偏差，不能简单归因于游戏画面整体拉伸。

近正方形压力测试结果：

- `1600×1600 / DPI 450` 会被 `image.width <= image.height` 横屏保护直接拒绝。
- `1601×1600 / DPI 450` 可以进入识别流程，战斗开始按钮最高得分约 `0.9045`。
- 加载模板最高仅约 `0.3152`，会话持续停在 `WAIT_LOADING`。
- 该分辨率下参考内容按宽度缩放成约 `1601×900`，上下各产生约 350px 留白；游戏加载图标却锚定整个屏幕右下角，当前统一居中的纵向映射无法覆盖。
- 当前发布目标仅覆盖约 `16:9～20:9` 横屏。近正方形画面被记录为异常窗口/折叠屏的已知限制，暂不增加 `TOP/BOTTOM` 纵向锚点。

## 已知限制与后续方向

当前限制：

1. 只有中央、右侧两种横向锚点。
2. 同一右侧偏移同时服务菜单、时钟、AUTO 和全局 SET。
3. 只校准水平位置，不校准垂直位置和 ROI 尺寸。
4. 时钟没有独立模板锚点校准。
5. 极端比例下，数字分位裁剪对少量像素偏差较敏感。
6. 竖屏帧会直接进入“等待游戏横屏”，不执行识别。
7. 接近正方形的横屏虽然可以绕过方向检查，但加载图标等屏幕边缘元素可能使用不同的纵向锚定规则，当前不保证支持。

后续改进方向：

1. 为时钟增加独立的二维偏移或局部搜索，不完全继承菜单偏移。
2. 记录完整时钟 ROI 和每个分位的实际矩形，便于定位边缘截断。
3. 在数字切分时允许少量水平滑动或可变分位宽度。
4. 检测游戏实际内容视口和安全区，而不是只根据整个截图尺寸推导。
5. 在 `WAIT_CLOCK` 状态同时显示 TP，并明确区分原始 OCR 与已接受时间。

## 主要代码位置

- `automation/GameCoordinateMapper.kt`：等比视口、中央/右侧锚点和校准偏移。
- `capture/ImageRoiExtractor.kt`：参考 ROI 到实际截图矩形的映射与像素提取。
- `capture/BattleReferenceRegions.kt`：所有识别区域的 `1920×1080` 参考坐标。
- `capture/FrameProcessor.kt`：战斗门控、模板校准、各识别器调用和状态发布。
- `capture/FixedTemplateMatcher.kt`：固定模板、尺度搜索和角色 SET 结构匹配。
- `recognition/ClockRecognizer.kt`：时钟分割、数字候选和置信度。
- `recognition/RecognitionFilter.kt`：时间合法性和时序过滤。
- `recognition/EnergyDetector.kt`：五名角色 TP 和 UB 触发检测。
- `control/BattleControlObservationFilter.kt`：AUTO/SET 跨帧稳定过滤。
- `control/ControlObservationSafetyGate.kt`：短暂不可信容忍和持续失败安全暂停。
