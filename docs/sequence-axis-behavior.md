# 顺序轴行为说明

顺序轴已进入 Android 原生运行链路，使用时钟调度、能量满到空的角色 UB 信号和画面控制状态确认执行动作。

## 核心语义

- `点击=角色N`不是永久切换，也不是简单地把当前状态取反。
- 节点触发后，工具确保角色N的 SET/立即发动状态为开。
- 工具等待该角色实际释放一次 UB。
- UB完成后，工具关闭该角色的 SET，使这次设置只生效一次。
- 完成上述生命周期后，才进入下一个顺序点击动作。
- TP 下降事件与时钟识别独立；即使该帧时钟识别失败，当前角色仍会立即进入 SET 关闭阶段。

同一时间包含多个角色时，必须保留原文顺序；每个角色都具有独立的“一次性 SET -> UB -> 关闭 SET”生命周期。AUTO点击仍按轴中出现顺序执行。

## 已支持

- `点击=角色1`～`点击=角色5`，以及轴头声明的角色别名；
- 同一行多个角色按从左到右顺序串行；
- `点击=AUTO` 的单次切换和画面确认；
- `AUTO=开/关`、完整五角色 `SET=...` 目标收敛；
- `提示=...`；
- `UB后=角色1`～`UB后=角色5`，节点到时后等待该角色下一次 UB 再执行本行动作；
- `UB后=BOSS`，检测实际 Boss UB 后立即执行；可选 `延迟=秒数` 再额外等待；
- `卡帧=角色1`～`卡帧=角色5`，进入手动逐帧并在确认时点击指定角色；
- 跳秒补执行，角色 UB 动画期间冻结普通定时调度；已检测到角色 UB 的 `UB后=角色N` 节点仍立即派发其后续动作；
- 悬浮窗显示当前生命周期阶段和下一动作。

## 触发节点格式

普通定时节点在时间到达后立即进入动作队列：

```text
1:20 | 点击=角色2
```

角色 UB 后节点在时间到达时开始监听，只接受节点进入等待状态之后出现的下一次对应角色 UB：

```text
1:20 | UB后=角色3 | 点击=角色2 | 提示=角色3 UB后执行角色2
```

已武装后的匹配 TP 事件会缓存到下一个可调度帧，不会因同帧时钟识别失败而丢失。

角色 TP 下降本身就是 `UB后=角色N` 的同步点，即使画面仍处于 UB 动画，后续 `点击=AUTO` 或角色动作也不会等待动画状态超时。

Boss UB 后节点到时只进入等待，不会仅因时间到达而执行。程序以倒计时异常停表为 Boss UB 候选，并以单角色 TP 下降排除角色 UB。长停表达到提前确认阈值后，无延迟节点会在 BOSS 动画结束附近立即执行；若声明正延迟，仍要等倒计时实际跳秒完成确认后才开始计时：

```text
0:56 | UB后=BOSS | 点击=角色5 | 点击=角色2
```

需要额外等待时可声明延迟，延迟从 Boss UB 检测时刻开始计算：

```text
0:30 | UB后=BOSS | 延迟=1.20 | 点击=AUTO
```

该方式不依赖固定分辨率图像模板。提前确认只针对无延迟 BOSS 节点，正延迟节点不会因提前事件提前计时；暂停菜单或采集间断会清除当前停表状态。提前确认时长可在“设置 → BOSS UB 提前确认”中按毫秒调整，默认 `7000ms`，允许 `3000～15000ms`，下一场战斗生效。

卡帧节点进入悬浮窗手动逐帧状态。点击“确定”时工具在已经打开的暂停菜单中点击指定角色，点菜单外返回战斗，然后自动执行该角色“等待 UB → 关闭 SET”生命周期，最后执行本行剩余动作：

```text
1:10 | 卡帧=角色4 | 提示=确认角色4动作帧
```

`卡帧=角色4`本身已经包含确认时点击角色4以及返回战斗后的“一次性 SET → UB → SET关闭”生命周期；同一行再次填写 `点击=角色4` 会去重。

同一节点不能同时声明`UB后`和`卡帧`。

## 当前限制

- `点击=BOSS` 当前只显示 Boss UB 提示，不代表 `UB后=BOSS` 的自动检测逻辑。
- 角色 UB 以对应角色能量条从满到低于阈值作为触发证据。若该证据丢失，角色生命周期会等待“顺序轴 SET 兜底”的独立 watchdog 到期后强制关闭 SET。
- watchdog 只在对应角色 SET 生命周期开始后武装；排队中未开始的角色不继承前一角色的截止点。
