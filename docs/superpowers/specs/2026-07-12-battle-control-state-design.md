# 战斗控制状态识别与开局收敛设计

日期：2026-07-12

## 目标

Android 助手必须从战斗画面识别 AUTO、全体立即发动以及角色 1～5 的立即发动状态，并将轴文件要求的开局状态可靠地收敛到游戏实际状态。状态不跨助手重启持久化；每场战斗都重新识别，以画面为唯一事实来源。

## 已确认的游戏规则

- 游戏会继承上一场战斗的 AUTO 和立即发动状态。
- 右侧 AUTO 按钮亮起表示 AUTO 开启。
- 右侧“立即发动”按钮亮起时，五名角色都处于立即发动状态。
- 点击亮起的全体立即发动按钮将其关闭，同时关闭五名角色的立即发动状态。
- 全体立即发动亮起时，点击单个角色只关闭该角色，并同时熄灭全体按钮；其他角色保持开启。
- 点击角色头像用于切换该角色的立即发动状态。
- 角色编号按画面从左到右为角色 1、2、3、4、5。

## 视觉样本

现有样本位于 `assets/templates/`：

- `auto_on.bmp`、`auto_off.bmp`：AUTO 开关样本。
- `set_on.bmp`、`set_off.bmp`：全体立即发动开关样本。
- `set.bmp`：单个角色开启立即发动后，头像右上角出现的蓝色“立即发动”徽标。
- `set_on_off_on_off_on.png`：角色 1、3、5 开启，角色 2、4 关闭，全体按钮关闭的 1920×1080 实战基准图。
- `menu.bmp`：右上角战斗菜单按钮文字，用于异常时确认菜单按钮存在并进入安全暂停。

按钮使用固定参考区域定位，但状态通过图像识别判断。所有参考区域以 1920×1080 为基准并随捕获画面缩放。AUTO 和全体按钮使用包含完整按钮的固定 ROI；五个角色徽标 ROI 分别位于五个头像右上角，中心横坐标约为 565、805、1045、1285、1525，纵坐标约为 795。最终整数 ROI 由基准图测试固化在 `BattleReferenceRegions` 中。

## 状态模型

```kotlin
enum class VisualToggleState { ON, OFF, UNKNOWN }

data class ToggleObservation(
    val state: VisualToggleState,
    val onScore: Double,
    val offScore: Double?,
    val margin: Double,
    val frameId: Long
)

data class BattleControlState(
    val auto: VisualToggleState,
    val globalSet: VisualToggleState,
    val roles: Map<CharacterRole, VisualToggleState>
)
```

当前战斗会话维护三份状态：

- `observedState`：最近一次经过一致性校验的画面状态。
- `desiredState`：轴文件当前要求达到的目标状态。
- `expectedState`：执行一个点击后，等待画面确认的预期状态。

另保存当前待确认动作、动作时间、确认帧数和重试次数。所有数据只存在于当前 `FrameProcessor`/战斗会话内；`prepareNewBattle()` 和进程重启都会清空。

控制流程另有 `RUNNING`、`SAFETY_PAUSING`、`SAFETY_PAUSED` 三种安全状态。`SAFETY_PAUSED` 只能由用户手动解除，助手不会自动点击菜单外区域恢复。

## 识别方法

### AUTO 与全体按钮

对同一个实际 ROI 分别计算开、关模板分数：

```text
onScore  = score(ROI, onTemplate)
offScore = score(ROI, offTemplate)
margin   = |onScore - offScore|
```

- 最佳分数达到最低阈值且分差达到最低边距时，输出对应的 `ON` 或 `OFF`。
- 分数或分差不足时输出 `UNKNOWN`。
- 模板尺寸可以不同；匹配器在同一实际 ROI 上做比例映射。

### 单角色立即发动

五个角色使用同一个 `set.bmp` 正模板：

- 徽标分数达到开启阈值时输出 `ON`。
- 徽标分数低于关闭阈值时输出 `OFF`。
- 位于两个阈值之间时输出 `UNKNOWN`。

基准截图中角色 1、3、5 是正样本，角色 2、4 是负样本。阈值必须由这些已知样本和后续实战诊断确定，不能仅凭目测设置。

### 一致性规则

- `globalSet == ON` 时，五个角色必须全部为 `ON`。
- 若全体按钮为 `ON`，但任一角色明确为 `OFF`，整组立即发动状态本帧无效，不更新 `observedState`。
- `globalSet == OFF` 时，五个角色允许任意混合。
- 任一必需状态为 `UNKNOWN` 时，不执行依赖该状态的收敛点击。

## 轴语义

- `点击=角色N`：无条件切换一次角色 N，随后进入视觉确认。
- `点击=AUTO`：无条件切换一次 AUTO，随后进入视觉确认。
- `AUTO=开/关`：根据真实观察状态收敛到目标值。
- `SET=开,关,开,关,开`：五个值固定对应角色 1～5，根据真实观察状态收敛。

不新增跨战斗配置文件，也不保存上一次确认状态。

## 开局状态处理

轴文件经常在 `1:30` 声明开局状态。由于 4 倍速或识别启动延迟可能使首个可信时钟落在 `1:29`～`1:28`，开局状态不能依赖 Scheduler 精确命中 90 秒。

轴加载时单独提取 90 秒事件中的 `AUTO=...` 和 `SET=...` 作为 `OpeningControlTarget`。战斗门禁在 88～90 秒锚定后：

1. 连续识别控制状态，建立 `observedState`。
2. 若轴没有开局控制目标，不做额外点击，立即进入正常调度。
3. 若存在目标，生成最少点击计划。
4. 每次只执行一个点击。
5. 设置 `expectedState`，等待连续两帧画面确认。
6. 确认后执行下一个点击。
7. 全部一致后标记 `openingControlsConfirmed=true`，再开放普通轴事件执行。

状态未确认期间 Scheduler 不执行可能与开局设置竞争的动作。

## 最少点击策略

### AUTO

- 观察值等于目标：不点击。
- 观察值与目标相反：点击 AUTO 一次。
- 观察值未知：等待，不点击。

### 角色 SET

- 目标全开：
  - 全体按钮已开：不点击。
  - 全体按钮已关且状态可信：点击全体按钮一次。
- 目标全关：
  - 全体按钮已开：点击全体按钮一次。
  - 全体按钮已关：仅点击仍为开的角色。
- 目标为混合状态：
  - 全体按钮已开：仅点击目标为关的角色；第一次角色点击会自动熄灭全体按钮。
  - 全体按钮已关：仅点击与目标不一致的角色。

每个点击完成并确认后重新计算计划，不能假设后续状态仍与旧计划一致。

## 点击确认与失败保护

- 点击后立即生成 `expectedState`，但不直接覆盖 `observedState`。
- 只有连续两帧识别结果与预期一致，才提交新状态并继续。
- 在 500 毫秒内未确认时，允许重试同一动作一次。
- 重试后仍不一致、控制状态连续不可信或出现规则矛盾时，进入 `SAFETY_PAUSING`。
- 未知状态必须 fail closed；历史状态和点击预期都不能替代明确的画面结果。

### 安全暂停

进入 `SAFETY_PAUSING` 后：

1. 立即冻结 Scheduler 和所有 AUTO/角色控制动作。
2. 在右上角菜单参考 ROI 匹配 `menu.bmp`。
3. 菜单可信时点击菜单按钮一次，游戏立即暂停，并进入 `SAFETY_PAUSED`。
4. 保存暂停原因、暂停前最后可信 observed/desired/expected 状态以及相关 ROI。
5. 菜单笼罩画面下所有常规识别结果都视为不可用，但不再累计新的识别错误，也不重复点击菜单。
6. 悬浮窗持续显示安全暂停原因，等待用户手动点击菜单外任意区域解除暂停。

助手不自动解除安全暂停。用户手动恢复后，菜单遮罩消失；识别器重新获得连续可信画面时清空 `SAFETY_PAUSED`，重新建立 `observedState` 并从当前目标重新规划，不能沿用暂停前尚未确认的点击预期。

若进入 `SAFETY_PAUSING` 时菜单模板也不可信，助手保持所有动作冻结并显示“无法确认菜单按钮”，不进行猜测坐标点击。

## 悬浮窗反馈

正常显示：

```text
AUTO:开  全体UB:关  角色:OXXXO
```

开局调整中：

```text
目标 AUTO:开 角色:OXXXO
当前 AUTO:关 角色:OOOOO  调整:AUTO
```

确认完成：

```text
AUTO:开 角色:OXXXO  开局已确认
```

失败示例：

```text
控制暂停：角色4预期开，实际为关
```

安全暂停示例：

```text
游戏已暂停：控制状态连续不可信
请手动点击菜单外区域恢复
```

## 诊断数据

诊断会话新增控制状态记录，至少包含：

- frameId、wallMs、门禁和时钟状态。
- AUTO 开/关分数、分差和最终状态。
- 全体立即发动开/关分数、分差和最终状态。
- 角色 1～5 徽标分数和最终状态。
- desired、expected、observed 状态。
- 点击动作、重试次数、确认结果和失败原因。

低置信、点击前、点击后和确认失败帧保存对应的小 ROI，不保存无界全屏截图。

## 组件边界

- `BattleControlRecognizer`：纯 Kotlin，输入各 ROI，输出视觉观察结果。
- `BattleControlStateMachine`：纯 Kotlin，维护 observed/desired/expected 和确认状态，生成单步动作。
- `BattleControlTemplates`：Android 资产加载。
- `SafetyPauseController`：确认菜单按钮、执行一次安全暂停并等待人工恢复。
- `FrameProcessor`：提取 ROI、调用识别器和状态机、控制开局门禁。
- `ActionExecutor`：只执行状态机批准的单个点击，不再自行猜测 AUTO 状态。
- `ControlDebugRecorder` 或现有诊断 Recorder 扩展：异步记录控制状态。

## 测试与验收

### 离线测试

- AUTO 与全体按钮四张模板能正确区分开/关。
- 混合截图准确输出角色 `O X O X O`。
- 全体按钮开启时校验五角色全开约束。
- 各种 observed/desired 组合生成最少点击集合。
- 全体按钮与单角色联动状态转换正确。
- 点击确认、超时、单次重试和 fail-closed 正确。
- 开场首个可信时间为 90～88 任一值时，90 秒开局目标都能执行一次且不重复。
- `prepareNewBattle()` 清空会话状态。
- 连续不可信、点击确认失败和状态矛盾都会进入安全暂停，菜单只点击一次。
- 菜单遮罩期间不累计新错误、不执行 Scheduler、不自动恢复。
- 用户手动恢复后丢弃未确认预期，以新画面重新规划。

### 模拟器验收

- 手动设置多种继承状态后开始新战斗，悬浮窗状态与画面一致。
- 使用 `AUTO=开` 和混合 `SET` 开局轴，最终状态与目标一致。
- 全开和全关使用全体按钮完成最少点击。
- 手动干预后下一帧以画面状态为准重新规划。
- 低置信或遮挡时不发生盲点。
- 制造状态识别错误时，助手点击菜单暂停；用户手动恢复后重新识别并继续。
